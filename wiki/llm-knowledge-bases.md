# LLM Wiki

利用 LLM 将零散的原始资料增量编译为结构化的 Markdown wiki，使知识持续复利而非每次查询时重新推导。由 Andrej Karpathy 于 2026 年 4 月提出，是对传统 RAG 模式的根本性替代。

## 核心思想：编译 vs. 检索

传统 RAG 的问题：每次查询都从零开始——上传文档，检索分块，生成回答。没有积累，没有综合，知识每次都被重新发现。NotebookLM、ChatGPT 文件上传等系统都是如此。

LLM Wiki 的不同之处：

- LLM **增量构建并维护一个持久的 wiki**——结构化、互相链接的 Markdown 文件集合
- 新增素材时，LLM 不是简单索引以备后用，而是读取、提炼、整合到现有 wiki 中
- 更新实体页面、修订主题摘要、标注新旧数据矛盾、强化或挑战已有综合
- **wiki 是一个持久的、复利式的产出物**——交叉引用已经建好，矛盾已经标出，综合已反映所有已读内容

> 人类的工作是策划来源、引导分析、提出好问题、思考意义。LLM 负责其余一切。

## 三层架构

### 1. Raw Sources（原始素材层）

精心策划的源文档集合：文章、论文、图片、数据文件。**不可变**——LLM 只读不改，这是事实来源。

### 2. Wiki（知识层）

LLM 生成的 Markdown 文件目录：摘要、实体页面、概念页面、对比、综述。LLM 完全拥有这一层——创建页面、在新素材到来时更新、维护交叉引用、保持一致性。人类阅读，LLM 编写。

### 3. Schema（规则层）

一份文档（如 `CLAUDE.md` 或 `AGENTS.md`），告诉 LLM wiki 的结构、约定和工作流。这是关键配置——使 LLM 成为有纪律的 wiki 维护者，而非通用聊天机器人。人类和 LLM 随使用共同演进这份文档。

## 核心操作

### Ingest（摄入）

将新素材放入 raw 目录，让 LLM 处理：

1. 读取素材，与用户讨论关键要点
2. 在 wiki 中写摘要页面
3. 更新索引
4. 更新 wiki 中相关的实体和概念页面
5. 在日志中追加条目

一个素材可能触及 10-15 个 wiki 页面。可以逐个摄入（深度参与）或批量摄入（少量监督）。

### Query（查询）

向 wiki 提问。LLM 搜索相关页面、阅读、综合带引用的回答。输出形式可以是 Markdown 页面、对比表格、Marp 幻灯片、matplotlib 图表等。

关键洞察：**好的回答可以作为新页面回流到 wiki 中。** 你的对比、分析、发现的关联——这些不应消失在聊天记录里。探索过程本身也在为知识库增值。

### Lint（健康检查）

定期让 LLM 对 wiki 做体检：

- 页面间的矛盾
- 被新素材取代的过时声明
- 没有入链的孤立页面
- 被提及但缺少独立页面的重要概念
- 缺失的交叉引用
- 可以通过网络搜索填补的数据空缺

## 索引与日志

### index.md（内容导向）

wiki 的目录——每个页面附链接、一行摘要和可选元数据，按分类组织。LLM 每次摄入后更新。查询时 LLM 先读索引找相关页面，再深入阅读。在中等规模（约 100 个来源、数百页面）下效果出奇地好，无需嵌入式 RAG 基础设施。

### log.md（时间导向）

只追加的时间记录——摄入、查询、lint 操作。格式如 `## [2026-04-02] ingest | Article Title`，可用简单工具解析。提供 wiki 演化的时间线，帮助 LLM 了解最近做过什么。

## 应用场景

- **个人**：追踪目标、健康、心理、自我提升——归档日记、文章、播客笔记
- **研究**：持续数周或数月的深度研究——阅读论文、文章、报告，增量构建综合 wiki
- **读书**：逐章归档，构建角色、主题、情节线索页面及其关联——类似个人版的 Tolkien Gateway
- **团队**：LLM 维护的内部 wiki，由 Slack 线程、会议记录、项目文档、客户通话喂养
- **竞品分析、尽职调查、旅行规划、课程笔记、兴趣深挖**——任何需要随时间积累知识的场景

## 推荐工具

- **Obsidian** 作为前端 IDE：浏览原始数据和编译后的 wiki，利用图谱视图观察 wiki 的形状
- **Obsidian Web Clipper**：浏览器扩展，将网页文章转为 Markdown
- **Marp**：基于 Markdown 的幻灯片格式，Obsidian 有插件
- **Dataview**：Obsidian 插件，对 frontmatter 运行查询，生成动态表格和列表
- **qmd**：本地 Markdown 搜索引擎，混合 BM25/向量搜索 + LLM 重排，支持 CLI 和 MCP server
- **Git**：wiki 就是 git 仓库，天然拥有版本历史、分支和协作能力

## 为什么有效

维护知识库的乏味部分不是阅读或思考——而是**簿记**：更新交叉引用、保持摘要最新、标注新旧矛盾、维护数十页面的一致性。人类放弃 wiki 是因为维护负担增长快于价值增长。LLM 不会无聊、不会忘记更新交叉引用、一次可以触及 15 个文件。wiki 得以维护是因为维护成本趋近于零。

这个思路精神上与 Vannevar Bush 的 Memex（1945）相关——一个私人的、策划的知识存储，文档间有关联轨迹。Bush 的愿景比后来的 Web 更接近这里的描述：私有的、主动策划的、文档间的连接与文档本身一样有价值。他无法解决的是谁来做维护。LLM 解决了这个问题。
